# cryptography_2

## Реалізація Струмок-256 та Струмок-512

Специфікація:

- Струмок-256: ключ 256 біт, ІV 256 біт, внутрішній стан 512 біт
- Струмок-512: ключ 512 біт, ІV 512 біт, внутрішній стан 1024 біт

`Внутрішній стан:
s[0..15] - LFSR (лінійний зсувний регістр, 16 × 64 біт)
r1, r2 - регістри нелінійної частини (FSM)`

`Параметри:
nwords - кількість 64-бітних слів для ключа та ІV
(4- Струмок-256, 8 -Струмок-512)`

### Завантаження ключа та ІV у внутрішній стан LFSR

Схема заповнення для Струмок-256 (nwords=4):
s[0..3] ← key[0..3]
s[4..7] ← iv[0..3]
s[8..11] ← key[0..3] (повторення)
s[12..15] ← iv[0..3] (повторення)

Для Струмок-512 (nwords=8):
s[0..7] ← key[0..7]
s[8..15] ← iv[0..7]

## Структура алгоритму:

- Лінійний зсувний регістр (LFSR): 16 слів по 64 біти
- Нелінійна частина (FSM): 2 слова-регістри (R1, R2)
- Нелінійне перетворення T (S-box + матриця MDS через попередньо обчислені таблиці)
- Множення на елементи α та α^-1 скінченного поля GF(2^64)

Крок LFSR відповідно до специфікації ДСТУ 8845:2019 :

    Характеристичний поліном над GF(2^64):
        s[16] = α·s[0] ⊕ s[2] ⊕ α⁻¹·s[15]
            = alpha_mul(s[0]) XOR s[2] XOR alphainv_mul(s[15])

    Після обчислення нового елемента стан зсувається:
        s[0..14] ← s[1..15], s[15] ← нове значення
        (deepseek написав красиво формули)

В реалізації коду біти переведено в байти для зручності використання обрахунків.

### Тестування реалізації шифру «Струмок»

1. (α та α⁻¹ — взаємно обернені)
2. Відтворюваність (однаковий ключ/ІV -> однакова гама)
3. Унікальність (різні ключі -> різні гами)
4. Симетрія (encrypt(decrypt(x)) == x)
5. Струмок-256 та Струмок-512 відрізняються для різних ключів
6. Референсні вектори (внутрішньо погоджені - для перевірки регресій)
7. Базова функціональність encrypt/decrypt

тестування реалізації:

```bash

python benchmark_strumok.py

```

## Порівняння «Струмка» з шифрами ZUC, SNOW 2.0, ChaCha20

- **ZUC** (стандарт 3GPP для 4G/LTE);
- **ChaCha20** (шифр від Daniel Bernstein, використовується в TLS, WireGuard, OpenSSH);
- **SNOW 3G** (попередник ZUC) та **SNOW 2.0** (прототип Струмок)

### Відмінності

#### Архітектура

- **Струмок** та **SNOW** використовують **LFSR + FSM** з операціями в скінченних полях
- **ChaCha20** базується на **ARX** (Add-Rotate-XOR)
- **ZUC** поєднує LFSR та нелінійну FSM, але з меншим розміром стану

#### Продуктивність

- На мові C: **Струмок-256** (14 Гбіт/с) випереджає ChaCha20 (7 Гбіт/с) та ZUC (1-2 Гбіт/с)
- На Python: ситуація зворотна - ChaCha20 працює швидше (20-50 МБ/с), оскільки ARX-операції легші для інтерпретатора, ніж табличні перетворення T та множення в GF(2^64)
- **Висновок:** Струмок оптимізований для апаратної реалізації та C/C++ (через таблиці), але Python-реалізація страждає через високі витрати

#### Безпека

- **Струмок-512** пропонує найбільший запас стійкості (2^512), що є надлишковим для практичних застосувань
- **ChaCha20** (2^256) вважається достатньо стійким для всіх сучасних застосунків
- **ZUC** (2^128) забезпечує мінімальний сучасний рівень безпеки

#### Стандартизація та впровадження

- **ChaCha20** є стандартом для інтернет-протоколів (TLS 1.3, WireGuard, OpenSSH)
- **ZUC** обов'язковий для 4G/LTE
- **Струмок** національний стандарт України, але поки що не набув широкого поширення за межами країни
